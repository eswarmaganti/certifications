# Amazon S3
- Amazon S3 is one of the main building blocks of AWS
- It's advertised as "infinitely scaling" storage
- Many websites use Amazon S3 as a backbone
- Many AWS services use Amazon S3 as an integration as well

### Amazon S3 use cases:
- Backup and storage
- Disaster Recovery
- Archive
- Hybrid Cloud Storage
- Application hosting
- Media hosting
- Data lakes & bid data analytics
- Software delivery
- Static Website

### Amazon S3 - Buckets
- Amazon S2 allow people to store objects (files) in "buckets" (directories)
- Buckets must have a globally unique name (across all regions all accounts)
- In March 2026, amazon released a new feature to create S3 buckets with unique names as per region which eliminates the overhead of global unique names for s3 buckets across all accounts and regions.
- Buckets are defined at region level
- S3 looks like a global service but buckets are created in a region
- Naming Conventions
  - No uppercase, No underscore
  - 3-63 chars long
  - Not an IP
  - Must start with lower case letter or number
  - Must NOT start with the prefix *xn--*
  - Must NOT end with the suffix *-s3alias*

### Amazon S3 - Objects
- Objects (files) have a Key
- The Key is the FULL path:
  - `s3://my-bucket/my_file.txt`
  - `s3://my-bucket/my_folder/another_folder/my_file.txt`
- The key is composed of prefix + object name
- There is no concept of "directories" within buckers
- Just keys with very long names that contain slashes "/".
- Objects values are the content of the body
  - Max Object size id 5TB (5000GB)
  - If uploading more than 5GB, must use "multi-part upload"
- Metadata 
- Tags
- Version ID (if versioning is enabled)

### Amazon S3 - security
- User Based
  - IAM policies - which API calls should be allowed for a specific user from IAM
- Resource Based
  - *Bucket Policies*: bucket wide rules from the s3 console - allows cross account 
  - *Object Access Control List (ACL)*: finer grain (can be disabled)
  - *Bucket Access Control List (ACL)*: less common (can be disabled)
- *Note*: an IAM principal can access an S3 object if
  - The user IAM permissions ALLOW it OR the resource policy ALLOWS it
  - AND there is no explicit DENY
- *Encryption*: encryption objects in Amazon S3 using encryption keys

### S3 Bucket Policy
- JSON based policies
  - Resources: buckets and objects
  - Effect: Allow/Deny
  - Actions: Set of API to allow or deny
  - Principal: The account or user to apply the policy to
- Use S3 bucket for policy to:
  - Grant public access to the bucket
  - Force objects to be encrypted at upload

### S3 bucket versioning
Amazon S3 bucket versioning is a data protection feature that enables you to store multiple variants of an object within the same bucket. It helps you recover from unintended user actions, such as accidental deletions or overwrites, by preserving every version of every object.

#### Key Concepts & Features
- **Data Protection**: Versioning provides an extra layer of safety, allowing you to restore previous versions of an object at any time, even after it has been deleted or overwritten.
- **Unique Version IDs**: When versioning is enabled, AWS S3 automatically assigns a unique version ID to each object version. Objects with the same key but different version IDs are stored in the bucket.
- **No Overwrites**: A new upload (PUT request) of an existing object does not replace the original instead, the new version is added, and the previous version is retained.
- **Delete Markers**: When an object is deleted,S3 does not permanently remote it. Instead, it inserts a "delete marker" as the current version. This hides the object from normal GET requests but keeps all previous versions available for restoration.
- **States**: An S3 bucker can be in one of three states:
  - **Unversioned**: The default state for a newly created bucket
  - **Enabled**: S3 saves multiple versions of an object
  - **Suspended**: Stops the creation of new object versions, but all existing versions remain in the bucket. Once versioning is enabled, it can never be returned to an "unversioned" state, only "suspended".
- **MFA Delete**: For added security on critical data, you can enable Multi-Factor Authentication Delete, which requires an MFA code to permanently delete an object version or change the bucket's versioning state.

### Amazon S3 - Replication CRR & SRR
Amazon S3 replication automatically copies objects between buckets for compliance, latency reduction or disaster recovery. Cross-Region Replication (CRR) duplicates data across different AWS Regions, which Same-Region Replication (SRR) operates within the same region.

#### Key Aspects of S3 Replication
- **CRR**: Designed for regional disaster recovery, complicance and reduced latency. Data is copied to a different geographic region.
- **SRR**: Used for log aggregation, replicating data between test and production, or complying within data sovereignty laws without transferring data out of the region.
- **Replication Features**:
  - **S3 RTC (Replication Time Control)**: Predictable replication for time-critical workloads
  - **S3 Batch Replication**: Used to replicate existing objects or previously failed replicas.
  - **Options**: Replicate to different accounts or the same accounts
- **Requirements**: The source and destination buckets must have versions enabled.

### S3 Storage Classes
Amazon S3 offers multiple storage classes designed for different data access patterns, costs and durability needs, all providing *99.999999999%* durability.

- **S3 Standard - General Purpose**: For frequently accessed data, offering high throughput and low latency
- **S3 Intelligent-Tiering**: Automatically moves data between two access tiers (frequent or infrequent ) to optimize costs without performance impact.
- **S3 standard IA**: Ideal for long-lived data that is accessed less frequently but requires immediate access, offering lower storage costs than Standard.
- **S3 One zone IA**: Stores data in a single Availability Zone, offering a 20% lower costs than Standard IA, suitable for non-critical, reproducible data.
- **S3 Glacier Instant Retrieval**: Provides low-cost storage for long-lived data rarely accessed, with millisecond retrieval times.
- **S3 Glacier Flexible Retrieval**: Used for data archiving with retrieval time ranging from minutes to hours.
- **S3 Glacier Deep Archive**: The lowest-cost storage class, suitable for data that is rarely accessed and retained for long periods (7-10) Years.
- **S3 Express One Zone**: Designed for high-performance, frequently accessed data, storing data within a single availability zone.

### S3 life cycle policy
An Amazon S3 lifecycle policy is a set of automated rules defining actions for managing objects, aimed at optimizing storage costs and efficiency. Policies include *transition actions* (moving data to cheaper storage classes like Glacier) and expiration actions (deleting old data or incomplete uploads). These rules can be applied to specific prefixes or tags.

#### Key components of S3 LifeCycle Rules
- **Rule Scope**: Applies to all objects, or filtered by prefix (e.g., `logs/`) or object tags
- **Transition Actions**: Moves objects to lower-cost stoage tiers (Standard -> Standard IA -> Glacier)
- **Expiration Actions**: Automatically deletes objects after a set of period, such as expiring objects after 30 days.
- **Versioned Object Management**: Managed non-current versions, allowing you to delete them after a specific number of days.
- **Incomplete Multipart Uploads**: Cleans up large, in-progress uploads that failed to complete, saving costs.

### AWS Snow Family
The AWS snow family is a collection os physical devices designed to securely move massive amounts of data into and out of the AWS Cloud. They are primarily used in environments with limited or no internet connectivity where traditional network transfers would be slow, expensive or impossible.

#### AWS Snowcone:
Snowcone is the smallest and most portable member, weighing about 4.5 lbs
- **Capacity**: Offers up to 8 TB (HDD) or 14TB (SSD) of usable storage
- **Best For**: Small-scale data migration and edge computing in remote or space-constrained environments like drones, military filed operations, or remote clinics.
- **Key Features**: Can be powered by a battery or a suitable USB-C power bank for mobile applications

#### AWS Snowball
Snowball is a suitcase-sized, ruggedized device designed for larger migrations and intensive local processing. It comes in two main versions:
- **Storage Optimized**: Focuses on high-capacity data transfer with up to 80TB or 210 TB of storage
- **Compute Optimized**: Designed for heavy local workloads, offering 52 vCOUs, higher memory, and optional NVIDIA GPUs for tasks like machine learning and video analytics at the edge.
- **Best For**: Migrating terabytes to petabytes of data or running local Amazon EC2 instances in disconnected location like oil rigs or ships

#### How the process works
1. **Request**: You order a device through the AWS Snow Family Management Console
2. **Ship**: AWS ships the device to your location
3. **Load**: You connect the device to your local network and copy your data onto it using tools like AWS OpsHub.
4. **Return**: You ship device back to AWS using the build-in E-ink shipping label.
5. **Upload**: AWS uploads your data to Amazon S3 and securely erases the device following NIST standards.
